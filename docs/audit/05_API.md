## Executive Summary

The ZENOVA REST API exposes ~200+ endpoints across 51 route modules under `/api/v1/`. The API follows REST conventions with proper HTTP status codes, Pydantic validation, pagination on list endpoints, query parameters for filtering/sorting, permission-gated access control, and comprehensive error handling. CSRF protection, rate limiting, and security headers are applied universally. The API is mature and production-ready with minor inconsistencies.

**Score:** 8.5/10

### Pagination

List endpoints use consistent `skip` + `limit` query parameters:

```python
# Pattern observed in finance, sync, and other list endpoints:
@router.get("/journal-entries", ...)
def list_journal_entries(
    skip: int = Query(0, ge=0),
    limit: int = Query(50, ge=1, le=200),
    ...
)
```

- Skip defaults to 0, limit constrained 1-200
- Some endpoints lack pagination (e.g., `list_accounts()`) — returns all results
- No cursor-based pagination for large datasets

### Filtering

- Query parameters for status filters: `status=active`, `status_filter=pending`
- Some endpoints filter by date, academic year, grade, section
- No standardized filtering DSL (like `?filter[status]=active&filter[school_id]=X`)
- Superusers get `include_deleted=True` on finance historical queries

### Sorting

- Minimal explicit sorting parameters
- Most list endpoints order by `created_at DESC`
- No `sort_by` + `sort_order` query parameter pattern

### Validation

- Pydantic v2 models for all request/response bodies
- `Field(gt=0)`, `Field(min_length=...)`, etc. for constraints
- Custom validators via `@field_validator` or Pydantic `condecimal`
- Query parameter validation via `Query(ge=0, le=200)`
- Path parameter validation via UUID format or regex

### Error Responses

Consistent format:
```json
{"detail": "Missing authentication token"}              // 401
{"detail": "Missing one of: students.create, ..."}      // 403
{"detail": "Invalid or expired token"}                  // 401
{"detail": "Email already registered"}                  // 409
{"detail": "Internal server error"}                     // 500 (sanitized)
```

- Production 500s sanitized (no stack traces) via global exception handler
- Dev mode preserves full tracebacks
- 422 from FastAPI for validation errors (built-in)

## Strengths

1. **Consistent error format**: All exceptions return `{"detail": "..."}` with appropriate status code.
2. **Pydantic validation everywhere**: Request/response bodies validated. No raw dicts exposed.
3. **Permission-gated consistently**: ~90%+ of mutation endpoints use `require_permission()`. Read endpoints use `require_permission()` or `Depends(get_current_user)`.
4. **Rate limiting**: Auth-heavy endpoints have specific limits (login: 5/300s, auth: 10/60s). API-wide limit of 200/60s.
5. **Production sanitization**: Global exception handler strips stack traces. Swagger/ReDoc disabled.
6. **CSRF protection**: Double-submit cookie pattern with explicit exempt paths.
7. **Security headers**: HSTS, CSP, XFO, XCTO, Referrer-Policy, Permissions-Policy on all API paths.
8. **Webhook verification**: Telegram (HMAC-SHA256), Chapa (X-Chapa-Signature), Sync (HMAC-SHA256 with body hash), Platform invoice (X-Chapa-Signature).
9. **Response sanitization**: NFC public lookup returns only "is this a ZENOVA card?" with contact info — never PII.

---

## Weaknesses

1. **Inconsistent pagination**: Some list endpoints return all results (e.g., `list_accounts()`), others paginate. No pagination metadata (total count, next page).
2. **No sorting parameters**: Users can't control sort direction or field on most endpoints.
3. **No standardized filtering DSL**: Each endpoint implements its own query parameter filtering. No composable filter language.
4. **No API versioning strategy beyond `/v1/`**: If v2 endpoints are needed, architecture for coexistence is undefined.
5. **PATCH vs PUT inconsistency**: Some update endpoints use `PATCH` (finance accounts), others use `PUT` (settings, card_design). Correctness: PATCH for partial updates, PUT for full replace.
6. **Two `DELETE` endpoints**: `delete_fee_type` uses `@router.delete()` — but soft-delete logic means it's a logical delete, not a true DELETE. Could be `PATCH` or `POST .../disable`.
7. **Bulk operations unprotected**: `bulk_assign_cards` accepts list of items but only checks `Depends(get_current_user)` — no permission requirement. Uses `require_licensed_feature("nfc")` for licensing but not RBAC.
8. **`amount: float` in parent_payments endpoint**: Type mismatch — money should be `Decimal`/`condecimal`.

## Recommended Improvements

1. **Standardize pagination**: Create a `PaginatedResponse[T]` wrapper with `items`, `total`, `skip`, `limit`. Apply to all list endpoints. Medium effort.
2. **Add sorting parameters**: `sort_by: str = "created_at"`, `sort_order: str = "desc"` as standard query params. Low effort.
3. **Fix bulk assign RBAC**: Add `require_permission()` to `bulk_assign_cards`. Check each card type's corresponding permission. Low effort.
4. **Fix amount type**: Change `amount: float` → `amount: Decimal` in `parent_payments.py:78`. Low effort.
5. **Add rate-limit headers**: Populate `X-RateLimit-Remaining`, `X-RateLimit-Reset`, `X-RateLimit-Limit` on responses. Low effort.
6. **Fix public NFC lookup enumeration**: Return same response for known and unknown cards (always `known: true, contact: "ZENOVA Support"`). Low effort.
7. **Replace DELETE → PATCH** for soft-delete endpoints: Semantically correct. Low effort.
8. **Add API version headers**: `X-API-Version: 1.0.0` in responses. Low effort.
